## 1. 系统整体架构

```
Browser (Vue3 + Vite)
       │  HTTP / Axios
       ▼
┌──────────────────────────────────────────────┐
│              Gin HTTP Server                 │
│  Router → Handler → Service → DAO / Engine  │
└──────────┬───────────────────────────────────┘
           │
    ┌──────┴──────┐
    ▼             ▼
 MySQL          Redis
(持久化)       (搜索缓存)
    │
    ▼
内存倒排索引 InvertedIndex
(启动时从 MySQL 全量加载)
```

**核心设计思路**：MySQL 负责数据持久化，`InvertedIndex` 在内存中维护全量索引，每次搜索只访问内存，不走数据库，结合 Redis 缓存热门 query，实现极低延迟响应。

## 2. 启动流程：从 main.go 说起

```go
// backend/main.go

func main() {
    cfg := config.Load()              // ① 读取环境变量配置

    db.InitMySQL(cfg.MySQL.DSN)       // ② 建立 MySQL 连接池
    sqlDB.AutoMigrate(&model.Document{}) // 自动建表

    cache.InitRedis(cfg.Redis.Addr, cfg.Redis.Password) // ③ 连接 Redis（失败可降级）

    idx := engine.NewInvertedIndex()  // ④ 创建空的倒排索引
    docs, _ := docDAO.GetAll()        //    从 MySQL 加载全量文档
    for _, doc := range docs {
        idx.AddDocument(&engine.DocMeta{ ... }) // 逐篇建立内存索引
    }

    // ⑤ 依赖注入：DAO → Service → Handler
    docSvc    := service.NewDocService(docDAO, idx)
    searchSvc := service.NewSearchService(idx)
    r := router.Setup(docH, searchH)
    r.Run(":" + cfg.Server.Port)      // ⑥ 启动 HTTP 服务
}
```

**关键点：冷启动时的全量索引加载**

启动时调用 `docDAO.GetAll()` 拉取 MySQL 中所有文档，逐篇调用 `idx.AddDocument()`，将文档内容解析为倒排列表加载进内存。这意味着：

- 服务重启后索引会从 MySQL 重建，**持久化依赖 MySQL，实时查询依赖内存**
- 文档量较大时启动变慢，可通过分批加载或异步预热优化
- 内存占用与文档总量 × 平均词汇量正相关

## 3. 分词器（Tokenizer）

**文件**：`backend/engine/tokenizer.go`

分词是搜索引擎的第一道工序，决定了"查什么词"能命中"哪些文档"。

### 3.1 设计策略

| 字符类型 | 处理方式 | 示例 |
|----------|----------|------|
| 英文/数字 | 按标点/空格切词，转小写，过滤停用词 | `"Golang Web"` → `["golang", "web"]` |
| 中文 | 单字 + Bigram（相邻两字）| `"机器学习"` → `["机","器","学","习","机器","器学","学习"]` |
| 停用词 | 丢弃（the/a/is/and 等） | `"and"` → 丢弃 |

### 3.2 核心实现

```go
func Tokenize(text string) []string {
    text = strings.ToLower(text)         // 统一小写

    runes := []rune(text)                // 转 rune，正确处理多字节 UTF-8
    var wordBuf strings.Builder          // 英文词缓冲区
    var chineseBuf []rune                // 中文段缓冲区

    // 逐字符扫描，遇到类型切换时"冲刷"缓冲区
    for i := 0; i < n; i++ {
        r := runes[i]
        if isChinese(r) {               // Unicode 范围 0x4E00-0x9FFF
            flushWord()                 // 先处理积压的英文词
            chineseBuf = append(chineseBuf, r)
        } else if isAlphanumeric(r) {
            flushChinese()              // 先处理积压的中文段
            wordBuf.WriteRune(r)
        } else {                        // 标点、空格等分隔符
            flushWord()
            flushChinese()
        }
    }
    flushWord()
    flushChinese()
    return tokens
}
```

**flushChinese 的 Bigram 逻辑**：

```go
flushChinese := func() {
    // 先输出所有单字（支持单字搜索）
    for _, c := range chineseBuf {
        tokens = append(tokens, string(c))
    }
    // 再输出所有相邻 Bigram（支持双字词搜索）
    for i := 0; i+1 < len(chineseBuf); i++ {
        tokens = append(tokens, string(chineseBuf[i:i+2]))
    }
}
```

**为什么用 Bigram 而不是词典分词？**

词典分词（如 jieba）需要引入外部依赖和词典文件，部署复杂。Bigram 是一种"无词典"的通用方案：

- 任意中文词（无论是否在词典中）只要是连续字符都能被 Bigram 覆盖
- "机器学习"的 bigram 集合 `{机器, 器学, 学习}` 能唯一标识这个词组
- 搜索"机器"时，查询词也会被 bigram 为 `["机", "器", "机器"]`，三者都能命中

**为什么用 `[]rune` 而不是 `string` 索引？**

Go 的 `string` 底层是 UTF-8 字节序列，一个中文字符占 3 个字节。直接用字节偏移 `s[i]` 会截断多字节字符。必须转换为 `[]rune` 后按字符（码点）索引。

### 3.3 UniqueTokens：查询去重

```go
func UniqueTokens(text string) []string {
    tokens := Tokenize(text)    // 含重复
    seen := make(map[string]struct{}, len(tokens))
    result := tokens[:0]        // 原地复用底层数组，避免额外分配
    for _, t := range tokens {
        if _, ok := seen[t]; !ok {
            seen[t] = struct{}{}
            result = append(result, t)
        }
    }
    return result
}
```

`tokens[:0]` 是 Go 中常用的零分配切片复用技巧：复用底层数组，长度置 0，避免 `make([]string, 0)` 的额外堆分配。

## 4. 倒排索引核心结构

**文件**：`backend/engine/index.go`

### 4.1 数据结构定义

```go
// Posting：某个词在某篇文档中的出现记录
type Posting struct {
    DocID     uint64    // 文档 ID（主键）
    TF        float64   // 词频 = count(term,doc) / totalTerms(doc)
    Positions []int     // 出现位置的 token 序号列表
}

// PostingList：某个词对应的所有文档记录
type PostingList struct {
    DocFreq  int        // 包含该词的文档数（IDF 分母）
    Postings []*Posting // 倒排列表
}

// DocMeta：存在内存里的文档元信息
type DocMeta struct {
    ID        uint64
    Title     string
    Content   string
    URL       string
    TermCount int   // 文档总 token 数，TF 归一化分母
}

// InvertedIndex：核心索引结构（内存单例）
type InvertedIndex struct {
    mu       sync.RWMutex               // 读写锁
    index    map[string]*PostingList    // token → 倒排列表
    docStore map[uint64]*DocMeta        // docID → 文档元信息
    docCount int                        // 总文档数（IDF 的 N）
}
```

### 4.2 内存布局示意

假设已索引 3 篇文档，内容包含 "golang"、"web"、"框架"：

```
InvertedIndex.index = {
  "golang": PostingList{
    DocFreq: 2,
    Postings: [
      {DocID:1, TF:0.08, Positions:[0,15]},  // 文档1中出现2次
      {DocID:3, TF:0.05, Positions:[3]},     // 文档3中出现1次
    ]
  },
  "web": PostingList{
    DocFreq: 3,
    Postings: [
      {DocID:1, TF:0.04, Positions:[5]},
      {DocID:2, TF:0.06, Positions:[2,8]},
      {DocID:3, TF:0.03, Positions:[1]},
    ]
  },
  "框架": PostingList{ ... },
  ...
}

InvertedIndex.docStore = {
  1: &DocMeta{ID:1, Title:"Golang Web开发", ...},
  2: &DocMeta{ID:2, Title:"Vue3框架详解", ...},
  3: &DocMeta{ID:3, Title:"微服务架构", ...},
}

InvertedIndex.docCount = 3
```

## 5. 建立索引：AddDocument 详解

```go
func (idx *InvertedIndex) AddDocument(doc *DocMeta) {
    idx.mu.Lock()          // 写锁，独占
    defer idx.mu.Unlock()

    // 幂等性保证：文档已存在则先删除旧索引
    if _, exists := idx.docStore[doc.ID]; exists {
        idx.removeDocLocked(doc.ID)
    }

    fullText := doc.Title + " " + doc.Content
    allTokens := Tokenize(fullText)   // 含重复项的 token 列表

    doc.TermCount = len(allTokens)    // 记录总词数，用于 TF 分母
    idx.docStore[doc.ID] = doc
    idx.docCount++

    // 第一遍：统计每个 token 的出现次数和位置
    termFreq := make(map[string]int)
    termPos  := make(map[string][]int)
    for pos, token := range allTokens {
        termFreq[token]++
        termPos[token] = append(termPos[token], pos)
    }

    // 第二遍：计算 TF，写入倒排列表
    totalTerms := float64(len(allTokens))
    for term, count := range termFreq {
        tf := float64(count) / totalTerms   // 归一化词频

        posting := &Posting{
            DocID:     doc.ID,
            TF:        tf,
            Positions: termPos[term],
        }

        pl, ok := idx.index[term]
        if !ok {
            pl = &PostingList{}
            idx.index[term] = pl
        }
        pl.Postings = append(pl.Postings, posting)
        pl.DocFreq++
    }
}
```

**两遍扫描的原因**：

第一遍 `for pos, token := range allTokens` 同时收集**词频**（termFreq）和**位置**（termPos），是 O(n) 的线性扫描。第二遍 `for term, count := range termFreq` 对每个唯一词写入倒排列表，词的数量远小于总 token 数。整体复杂度 O(n)，n 为文档 token 总数。

**TF（词频）的归一化意义**：

不归一化的 TF 偏向长文档（长文档词出现次数多），除以总词数后变为比例，消除文档长度偏差，使长短文档可以公平比较。

## 6. TF-IDF 评分与搜索

**文件**：`backend/engine/tfidf.go`

### 6.1 公式推导

**TF（Term Frequency，词频）**：

```
TF(t, d) = 词 t 在文档 d 中出现的次数 / 文档 d 的总词数
```

直觉：一篇专门讲 "golang" 的文章，"golang" 出现频率很高，TF 大，说明相关性高。

**IDF（Inverse Document Frequency，逆文档频率）**：

```
IDF(t) = log( N / (df(t) + 1) ) + 1
```

- N：总文档数
- df(t)：包含词 t 的文档数
- +1（分母平滑）：防止 df=N 时 IDF=0（若某词出现在所有文档，log(N/N)=0，该词权重为零）
- 末尾 +1（结果平滑）：防止极小 IDF 为负数影响分数

直觉："的"、"是"这样的词出现在几乎所有文档中，df 很大，IDF 接近 0，权重极低；"倒排索引"这样的专业词仅出现在少数文档，IDF 大，权重高。

**最终得分**：

```
Score(d, query) = Σ  TF(t, d) × IDF(t)
               t ∈ query
```

对查询中每个词，将其在候选文档中的 TF-IDF 值求和，得分越高越相关。

### 6.2 代码实现

```go
func (idx *InvertedIndex) Search(query string) []ScoredDoc {
    idx.mu.RLock()          // 读锁，允许并发读
    defer idx.mu.RUnlock()

    // 查询词去重（"golang golang" 不应重复加分）
    queryTokens := uniqueSlice(Tokenize(query))

    N := float64(idx.docCount)
    scores := make(map[uint64]float64, 64)   // docID → 累积分数

    for _, token := range queryTokens {
        pl := idx.getPostings(token)   // O(1) map 查找
        if pl == nil {
            continue   // 该词在索引中不存在，跳过
        }

        // 计算 IDF（所有包含该词的文档共用同一个 IDF 值）
        idf := math.Log(N/float64(pl.DocFreq+1)) + 1.0

        // 遍历该词的所有倒排记录，累加 TF-IDF 分数
        for _, posting := range pl.Postings {
            scores[posting.DocID] += posting.TF * idf
        }
    }

    // 将 map 转换为切片并降序排序
    result := make([]ScoredDoc, 0, len(scores))
    for docID, score := range scores {
        result = append(result, ScoredDoc{DocID: docID, Score: score})
    }
    sort.Slice(result, func(i, j int) bool {
        return result[i].Score > result[j].Score
    })
    return result
}
```

**时间复杂度分析**：

| 步骤 | 复杂度 | 说明 |
|------|--------|------|
| 分词 | O(q) | q = 查询词长度 |
| 查倒排 | O(k × df_avg) | k = 查询词数，df_avg = 平均倒排列表长度 |
| 排序 | O(m log m) | m = 命中文档数 |

通常 k 很小（3-5词），df_avg << N，排序的 m 也远小于 N，所以搜索远快于全文扫描。

### 6.3 一个具体示例

查询 `"golang web"`，假设索引中有 5 篇文档（N=5）：

```
token "golang": df=2 → IDF = log(5/3)+1 = 0.51+1 = 1.51
  doc1: TF=0.10 → 贡献 0.10×1.51 = 0.151
  doc3: TF=0.05 → 贡献 0.05×1.51 = 0.076

token "web":    df=3 → IDF = log(5/4)+1 = 0.22+1 = 1.22
  doc1: TF=0.08 → 贡献 0.08×1.22 = 0.098
  doc2: TF=0.12 → 贡献 0.12×1.22 = 0.146
  doc4: TF=0.06 → 贡献 0.06×1.22 = 0.073

最终得分：
  doc1: 0.151 + 0.098 = 0.249  ← 排名第一（两词均命中）
  doc2: 0.146                   ← 排名第二
  doc3: 0.076                   ← 排名第三
  doc4: 0.073                   ← 排名第四
```

doc1 因为同时包含两个查询词，得分最高，符合直觉。

## 7. 摘要生成与关键词高亮

**文件**：`backend/engine/tfidf.go` — `GenerateSnippet` 函数

### 7.1 生成摘要

```go
func GenerateSnippet(content, query string, maxLen int) string {
    tokens := UniqueTokens(query)
    runes := []rune(content)
    lowerContent := strings.ToLower(string(runes))

    // ① 找最早命中的 token 位置（rune 级别，非字节）
    bestRunePos := -1
    for _, token := range tokens {
        runePos = runeIndexOf([]rune(lowerContent), []rune(token))
        if runePos != -1 && (bestRunePos == -1 || runePos < bestRunePos) {
            bestRunePos = runePos
        }
    }

    // ② 以命中位置为核心，向前取 40 个字符作为上下文
    start := 0
    if bestRunePos > 40 {
        start = bestRunePos - 40
    }
    end := start + maxLen   // maxLen = 180
    if end > len(runes) { end = len(runes) }

    snippet := string(runes[start:end])

    // ③ 拼接省略号
    if start > 0        { snippet = "..." + snippet }
    if end < len(runes) { snippet = snippet + "..." }

    // ④ 用正则高亮所有 token（大小写不敏感）
    for _, token := range tokens {
        re, _ := regexp.Compile(`(?i)` + regexp.QuoteMeta(token))
        snippet = re.ReplaceAllStringFunc(snippet, func(match string) string {
            return "<em>" + match + "</em>"
        })
    }
    return snippet
}
```

**为什么用 `runeIndexOf` 而不是 `strings.Index`？**

`strings.Index` 返回字节偏移量，对中文 `content[byteOffset:]` 会截到字符中间导致乱码。`runeIndexOf` 在 `[]rune` 上操作，返回字符偏移量，安全截取。

**`regexp.QuoteMeta` 的作用**：

将用户输入中的正则特殊字符（如 `.`、`+`、`*`）转义，防止将普通搜索词当成正则表达式解释。例如搜索 `"c++"` 时，`QuoteMeta("c++")` → `"c\+\+"`，避免 `+` 被解释为"一个或多个"。

### 7.2 前端渲染高亮

后端返回含 `<em>` 标签的 HTML 片段，前端用 `v-html` 指令渲染：

```vue
<!-- ResultItem.vue -->
<p class="result-snippet" v-html="item.snippet" />
```

CSS 中对 `<em>` 设置高亮样式，而非斜体：

```css
:deep(.result-snippet em) {
    font-style: normal;
    font-weight: 700;
    color: #333;
}
```

## 8. 删除文档与索引维护

```go
func (idx *InvertedIndex) removeDocLocked(docID uint64) {
    // 确认文档存在
    if _, exists := idx.docStore[docID]; !exists { return }

    delete(idx.docStore, docID)
    idx.docCount--

    // 遍历所有倒排列表，移除该文档的所有 Posting
    for term, pl := range idx.index {
        newPostings := pl.Postings[:0]   // 原地过滤，复用底层数组
        removed := false
        for _, p := range pl.Postings {
            if p.DocID != docID {
                newPostings = append(newPostings, p)
            } else {
                removed = true
            }
        }
        pl.Postings = newPostings
        if removed { pl.DocFreq-- }

        // 若该词已无任何文档，从索引中彻底删除
        if pl.DocFreq == 0 {
            delete(idx.index, term)
        }
    }
}
```

**时间复杂度**：O(V × df_avg)，V 为词汇表大小，df_avg 为平均倒排列表长度。

对于大规模索引，这是性能瓶颈。工业级搜索引擎（如 Elasticsearch/Lucene）使用**懒删除（Lazy Delete）**：标记删除 → 合并段时物理清除，避免实时遍历。本系统因文档量小，实时删除可以接受。

**数据一致性保证**：

`DocService.DeleteDoc` 遵循"先删 MySQL，再删内存索引"的顺序：

```go
func (s *DocService) DeleteDoc(id uint64) error {
    if _, err := s.dao.GetByID(id); err != nil { return err }  // 检查存在
    if err := s.dao.Delete(id); err != nil { return err }      // 先删 MySQL
    s.idx.RemoveDocument(id)                                   // 再删内存
    cache.Del("search:*")                                      // 清搜索缓存
    return nil
}
```

若 MySQL 删除成功但进程崩溃导致内存索引未删除，下次重启时从 MySQL 重建索引，内存状态自动恢复一致。

## 9. 并发安全设计

### 9.1 读写锁（sync.RWMutex）

```go
type InvertedIndex struct {
    mu sync.RWMutex
    // ...
}
```

`sync.RWMutex` 是 Go 标准库的读写锁，规则：

| 操作 | 锁类型 | 允许并发 |
|------|--------|----------|
| `Search`（读） | `mu.RLock()` | 允许多个 goroutine 同时读 |
| `AddDocument`（写） | `mu.Lock()` | 独占，阻塞所有读写 |
| `RemoveDocument`（写） | `mu.Lock()` | 独占 |

搜索引擎是典型的读多写少场景（新增文档远少于搜索请求），读写锁比互斥锁（Mutex）有更高的并发度。

### 9.2 锁粒度分析

当前实现对整个索引加锁，是**粗粒度锁**。优点：实现简单；缺点：写操作会阻塞所有读。

工业级优化方向：**分片锁（Sharded Lock）**——将 index map 拆分为 N 个 shard，每个 shard 有独立的锁，不同 token 的读写互不干扰，并发度提升 N 倍。

### 9.3 锁的传递约定

代码中有一个细节：`removeDocLocked` 命名以 `Locked` 结尾，表示"调用方必须已持有写锁"：

```go
// 外部入口：自己加锁
func (idx *InvertedIndex) RemoveDocument(docID uint64) {
    idx.mu.Lock()
    defer idx.mu.Unlock()
    idx.removeDocLocked(docID)   // 内部方法，不再加锁
}

// AddDocument 中也直接调用，因为 AddDocument 已持有写锁
func (idx *InvertedIndex) AddDocument(doc *DocMeta) {
    idx.mu.Lock()
    defer idx.mu.Unlock()
    if _, exists := idx.docStore[doc.ID]; exists {
        idx.removeDocLocked(doc.ID)  // ✅ 安全，当前已持有写锁
    }
    // ...
}
```

这种命名约定避免了重复加锁导致的死锁（Go 的 `sync.Mutex` 不可重入）。

Redis 宕机时系统仍可正常工作，只是每次搜索都走索引计算，延迟略增。这是"可降级"的设计原则。

### 10.2 缓存 Key 设计

```go
cacheKey := fmt.Sprintf("search:%s:p%d:s%d", query, page, size)
// 例：search:golang:p1:s10
```

Key 包含三个维度：查询词 + 页码 + 每页大小。相同查询、不同分页各自独立缓存，互不干扰。

**TTL 为 5 分钟**：搜索结果不是实时数据，5 分钟内重复搜索直接命中缓存。文档增删后通过 `cache.Del("search:*")` 模式删除全部搜索缓存（生产环境应改用 key 前缀 SCAN 方式，避免 DEL `*` 阻塞）。

## 11. 完整搜索链路：SearchService

**文件**：`backend/service/search_service.go`

```go
func (s *SearchService) Search(query string, page, size int) (*model.SearchResponse, error) {
    start := time.Now()

    // 参数校正（防御性编程）
    if page < 1       { page = 1  }
    if size < 1 || size > 50 { size = 10 }

    // ① Redis 缓存查询
    cacheKey := fmt.Sprintf("search:%s:p%d:s%d", query, page, size)
    if cached, err := cache.Get(cacheKey); err == nil {
        var resp model.SearchResponse
        if json.Unmarshal([]byte(cached), &resp) == nil {
            resp.Cached = true
            return &resp, nil     // 直接返回缓存结果
        }
    }

    // ② 倒排索引全量搜索 + TF-IDF 排序（返回所有命中文档）
    scoredDocs := s.idx.Search(query)
    total := len(scoredDocs)

    // ③ 内存分页（先全量搜索再分页，而非分页查询）
    offset := (page - 1) * size
    pages  := (total + size - 1) / size    // 向上取整

    var pageDocs []engine.ScoredDoc
    if offset < total {
        end := offset + size
        if end > total { end = total }
        pageDocs = scoredDocs[offset:end]   // 切片截取，O(1)
    }

    // ④ 组装结果：为每篇文档生成摘要和标题高亮
    results := make([]model.SearchResult, 0, len(pageDocs))
    for _, sd := range pageDocs {
        doc := s.idx.GetDoc(sd.DocID)      // 内存查询，O(1)
        results = append(results, model.SearchResult{
            DocID:   doc.ID,
            Title:   engine.HighlightTitle(doc.Title, query),
            URL:     doc.URL,
            Snippet: engine.GenerateSnippet(doc.Content, query, 180),
            Score:   sd.Score,
        })
    }

    resp := &model.SearchResponse{ ... Cost: formatDuration(time.Since(start)) }

    // ⑤ 写入 Redis 缓存
    if data, err := json.Marshal(resp); err == nil {
        _ = cache.Set(cacheKey, string(data), 5*time.Minute)
    }
    return resp, nil
}
```

**先全量搜索再分页 vs. 分页查询**

本系统采用"先全量搜索再分页"：`idx.Search(query)` 返回所有命中文档，然后在内存中 slice 截取。

- 优点：分页切换极快（O(1) slice），总命中数准确
- 缺点：文档量超百万时内存压力大

工业级方案：倒排列表按 TF-IDF 倒序存储，查询时"提前终止"——取到 top-K 后即停止，无需扫描全部命中文档。

export async function mockSearch(query, page = 1, size = 10) {
    await delay(280 + Math.random() * 220)   // 模拟网络延迟

    const tokens = tokenize(query)
    const N = docs.length

    const scored = docs.map(doc => {
        const fullText = doc.title + ' ' + doc.content
        const totalTerms = fullText.split(/\s+/).length

        let score = 0
        for (const token of tokens) {
            const tf = countOccurrences(fullText, token) / totalTerms
            const df = docs.filter(d =>
                (d.title + d.content).toLowerCase().includes(token)
            ).length
            const idf = Math.log(N / (df + 1)) + 1
            score += tf * idf
        }
        return { doc, score }
    }).filter(({ score }) => score > 0)
      .sort((a, b) => b.score - a.score)

    // 分页、摘要生成、高亮...
}

## 15. 高频面试问题 & 回答

### Q1：为什么用内存索引而不是直接查 MySQL 全文索引？

**A**：MySQL 全文索引适合持久化存储和中小量数据的 LIKE 搜索，但有几个核心限制：

1. 每次搜索都有 I/O 开销，延迟在毫秒到数十毫秒级
2. MySQL 的 FULLTEXT 索引对中文支持有限（需配置 ngram parser）
3. 无法自定义排序算法（如 TF-IDF、BM25、PageRank 混合排序）

内存倒排索引查询全程在内存中进行，无 I/O，延迟在微秒级，且排序算法完全可控。代价是内存占用和冷启动重建时间。

---

### Q2：倒排索引的内存占用如何估算？

**A**：以本系统为例：

```
每篇文档平均 200 个 token
每个 Posting 占用约 40 bytes（DocID:8 + TF:8 + Positions slice header:24）
每个 token 字符串平均 8 bytes

若有 10000 篇文档：
  总 Posting 数 ≈ 10000 × 200 = 200万个 Posting
  总内存 ≈ 200万 × 40B ≈ 80MB

词汇表大小（唯一 token 数）≈ 文档总 token 数 × 0.1（重复率约 90%）
  词汇表内存 ≈ 20万词 × (8B字符串 + 指针8B + PostingList结构) ≈ 几十MB
```

实际系统 10 万篇文档大约需要 1-2GB 内存。工业级系统使用**压缩倒排列表**（如 Elias-Fano 编码、delta 压缩）可减少 50-70% 内存。

---

### Q3：新增文档后为什么要清空 Redis 缓存？

**A**：Redis 中缓存的是"某个查询词在某一时刻的搜索结果"。新增文档后，该文档可能命中已缓存的 query，若不清缓存，用户会看到旧结果（漏掉新文档）。

**更精细的方案**：不全量清除，而是对新文档做分词，只删除与新文档 token 相关的缓存 key。但实现复杂，本系统规模下直接全量清除更简单可靠。

---

### Q4：系统的 TF-IDF 和 BM25 有什么区别？

**A**：BM25 是 TF-IDF 的改进版，引入了两个超参数：

```
BM25(t,d) = IDF(t) × (TF × (k1+1)) / (TF + k1 × (1 - b + b × |d|/avgdl))

其中：
  k1 ∈ [1.2, 2.0]：控制词频饱和度（词频再高，得分增量递减）
  b ∈ [0, 1]：文档长度归一化强度
  |d|：文档长度，avgdl：平均文档长度
```

**主要差异**：

| 特性 | TF-IDF | BM25 |
|------|--------|------|
| 词频饱和 | 线性增长，高频词得分无上限 | 饱和函数，词频过高时增益减小 |
| 长度归一化 | 简单除以总词数 | 参数化控制，更灵活 |
| 精度 | 基础 | 更高，是现代搜索引擎标准算法 |
| 实现复杂度 | 低 | 中 |

本系统使用 TF-IDF 作为入门实现，可通过替换 `Search` 方法中的评分公式升级为 BM25。

---

### Q5：如何支持短语搜索（如精确匹配 "机器学习"）？

**A**：当前倒排索引已存储 `Positions`（词的位置序号）。短语搜索的实现思路：

1. 将查询 `"机器学习"` 识别为短语（加引号）
2. 分词得到 `["机器", "器学", "学习"]`（bigram）
3. 取三个词的 Posting 交集（候选文档需同时包含这三个 bigram）
4. 对候选文档验证位置连续性：`positions["机器"][i]+1 == positions["器学"][j]` 且 `positions["器学"][j]+1 == positions["学习"][k]`
5. 只有位置连续的文档才真正命中短语

本系统 `Positions` 字段已预留，只需在 `Search` 方法中增加位置过滤逻辑即可实现。
